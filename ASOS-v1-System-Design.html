AI Sales
Operating System

Multi-tenant SaaS that converts Meta Ads traffic into WhatsApp conversations, qualifies leads via Claude AI, and closes deals — with full CRM, analytics, and white-label support for multiple businesses.

Multi-tenant Node.js + PostgreSQL WhatsApp Cloud API Claude AI (Anthropic) Meta Ads API Redis Queue Row-Level Security
01
System Architecture
High-level data and control flow across all layers
Layer 1 — Traffic Sources
📢
Meta Ads
Facebook / Instagram ads with Click-to-WhatsApp CTA. Carries UTM params + ad_id, adset_id, campaign_id.
🔗
WhatsApp Deep Link
wa.me link with pre-filled message encoding the campaign source. Routes user to Business number.
📊
Meta Conversions API
Server-side pixel events. Fires lead, purchase events back to Meta for attribution + optimization.
↓   ↓   ↓Webhook / API Call
Layer 2 — API Gateway & Webhook Ingestion
🚪
API Gateway
Reverse proxy (Nginx/Caddy). Rate limiting, SSL termination, tenant routing via subdomain or header.
🔔
WhatsApp Webhook
Receives inbound messages from Meta Cloud API. Verifies HMAC signature. Publishes to message queue.
🔐
Auth Service
JWT + refresh tokens. Tenant-scoped claims. Role-based: superadmin, tenant_admin, agent.
↓   ↓   ↓Internal Event Bus
Layer 3 — Application Core (Node.js Microservices)
💬
Conversation Service
State machine for each conversation. Orchestrates AI responses, human handoff, stage transitions.
🤖
Dual AI Engine (v1.5)
Qualifier AI scores intent → Closer AI generates reply. Both return strict JSON. Audit logged to AiAgentLog.
👥
CRM Service
Central record system. Manages leads, contacts, deals, activities, stages and scores.
📣
Campaign Service
Links Meta Ad campaigns to ASOS tenants. Tracks spend, impressions, conversions.
📈
Analytics Service
Aggregates funnel metrics, AI performance, ROI. Powers dashboard charts.
🔔
Notification Service
Alerts agents on hot leads, escalations, AI failures. Email + in-app.
↓   ↓   ↓Read / Write
Layer 4 — Data Layer
🐘
PostgreSQL
Primary data store. Row-Level Security enforces tenant isolation. All CRM, billing, config data.
⚡
Redis
Session cache, conversation state cache, job queue (BullMQ), rate limit counters.
📁
Object Storage
WhatsApp media (images, audio, docs). S3-compatible. CDN-served per tenant.
🔍
Vector Store
Tenant product knowledge base for Claude RAG. pgvector extension on PostgreSQL.
Multi-tenancy model: Shared database, shared schema, tenant isolation via tenant_id FK on every table + PostgreSQL Row Level Security (RLS) policies. Each tenant gets a dedicated WhatsApp Business number, Meta Pixel, and AI prompt configuration. Superadmin has unscoped access for billing and support.

Backend Architecture
Node.js services, middleware layers, and infrastructure
Core API Server
Express.js · Port 3000
  • REST API router with versioning (/api/v1)
  • Tenant resolution middleware (X-Tenant-ID or subdomain)
  • JWT auth + RBAC middleware
  • Request validation (Zod schemas)
  • Error handler + structured logging (Pino)
  • OpenAPI / Swagger docs generation
Webhook Worker
Express.js · Port 3001
  • POST /webhooks/whatsapp — receives WA Cloud API events
  • GET /webhooks/whatsapp — WA verification handshake
  • POST /webhooks/meta — receives Conversions API callbacks
  • HMAC-SHA256 signature verification
  • Idempotency check (dedup via Redis)
  • Publishes to BullMQ "messages" queue
Conversation Engine
BullMQ Worker Process
  • Consumes "messages" queue jobs
  • Loads conversation state from Redis cache
  • Determines: AI response | human agent | auto-reply
  • Calls Claude AI Engine service
  • Sends reply via WhatsApp Cloud API
  • Updates CRM lead record + conversation log
  • Fires Conversions API event to Meta
Claude AI Engine
Internal Service Module
  • v1.5 Dual Agent — Qualifier (haiku, temp 0) + Closer (sonnet, temp 0.4)
  • Loads tenant system prompt from DB
  • Fetches RAG context from pgvector
  • Builds message array with full conversation history
  • Qualifier returns {lead_status, score 1-10, intent, problem_summary, next_action}
  • Closer returns {reply_message, closing_type, urgency_trigger}
  • Both outputs logged to ai_agent_logs for Analyst AI (v2)
  • Score ≥ 8 → humanFollowupRequired=true; AI keeps replying
  • Logs token usage per tenant for billing
CRM Service
Internal Service Module
  • Lead CRUD + stage pipeline management
  • Contact deduplication (phone + email)
  • Deal value + probability scoring
  • Activity timeline logging
  • Agent assignment logic
  • Tag + segment management
Scheduler Service
Node-cron · BullMQ Delayed
  • Follow-up message scheduling
  • Stale lead re-engagement triggers
  • Daily analytics aggregation jobs
  • Meta Conversions API batch flush
  • Subscription billing cycle checks
  • Tenant health monitoring
Billing Service
Stripe Integration Module
  • Subscription plan management (Free, Pro, Enterprise)
  • Usage metering (AI tokens, contacts, messages)
  • Stripe webhook handler
  • Invoice generation
  • Feature flag enforcement by plan
  • Trial period management
Analytics Service
Internal Service Module
  • Funnel stage conversion rates
  • AI qualification accuracy metrics
  • Revenue attribution by ad campaign
  • Agent performance reports
  • WhatsApp message delivery rates
  • Cost per lead / cost per close

06
WhatsApp + Claude AI Integration Flow
End-to-end flow from ad click → AI response → CRM update → Meta attribution
1
Ad Click → WhatsApp Entry
User clicks a Click-to-WhatsApp Meta Ad. The WA deep link pre-fills a message containing an encoded campaign identifier (e.g. "Quero saber mais [src:camp_2024_promo]"). This lands the user in a conversation with the tenant's WhatsApp Business number.
📢
Meta Ad fires Pixel eventAd click event logged client-side. UTM params + ad_id + adset_id + campaign_id captured in WA entry message payload.
📱
WhatsApp message createdWhatsApp Cloud API fires a webhook to our server: POST /webhooks/whatsapp with the full message object including the user's WA ID (phone number) and message text.
2
Webhook Ingestion + Deduplication
The Webhook Worker receives the event, verifies the HMAC-SHA256 signature from Meta, checks Redis for duplicate wa_message_id, and publishes a job to the BullMQ "messages" queue.
🔐
Signature verificationX-Hub-Signature-256 header compared against HMAC of raw body using tenant's WA App Secret.
🔁
Idempotency checkRedis key: wa:dedup:{wa_message_id} with 24h TTL. Prevents double-processing on WA retry deliveries.
📨
Queue publishJob added to BullMQ queue with payload: {tenantId, phone, messageText, waMessageId, adSource, timestamp}.
3
Contact + Lead Resolution
The Conversation Worker picks up the job. It resolves (or creates) the contact and lead records in the CRM before invoking AI.
🔍
Contact lookupQuery contacts WHERE phone = E.164(phone) AND tenant_id = X. If not found, create new contact with WA profile name + phone.
🎯
Lead resolutionCheck for open lead linked to contact. If none, create new lead with stage = "new", linking to campaign via decoded ad source. Parse UTM + meta_ad_id from message payload.
💬
Conversation stateLoad or create conversation record. Check Redis for cached conversation state: {stage, messageHistory[], aiEnabled}. If cache miss, hydrate from DB (last 30 messages).
4
Claude AI Processing
The AI Engine builds a rich context and sends it to Claude. It parses the structured response to drive CRM state transitions and WhatsApp replies.
📝
Prompt assemblySystem prompt = tenant's custom persona + qualification criteria + closing script + language + current date. RAG context fetched from pgvector (cosine similarity search on message embedding).
📚
Message history injectionLast N conversation turns injected as messages array: [{role: "user", content: ...}, {role: "assistant", content: ...}]. Latest inbound message appended as final user turn.
🤖
Claude API callModel: claude-3-5-sonnet-20241022. Response format: JSON with schema {reply: string, stage: LeadStage, ai_score: 0-100, qualification_data: object, action: "continue" | "close" | "handoff", handoff_reason?: string}.
🧠
Structured output parsingJSON response validated against Zod schema. On parse failure: retry once with explicit format reminder. On second failure: log error, send safe fallback reply, flag for human review.
5
Action Routing (Continue | Handoff | Close)
Based on Claude's action field, the worker routes to one of three paths.
▶️
action: "continue"Send AI reply via WhatsApp Cloud API. Update conversation state in Redis. Persist inbound + outbound messages to DB. Update lead stage + ai_score if changed.
🙋
action: "handoff"Set conversation.status = "human_takeover". Disable AI for conversation. Notify assigned agent via in-app + email notification. Send fallback message to user: "Um especialista vai te atender em breve ✅". Log handoff_reason in activity feed.
🏆
action: "close"AI detected buying signal. Set lead.stage = "closed_won" or presents closing offer. Creates follow-up task for agent. Fires "Purchase" event to Meta Conversions API for ROAS measurement.
6
WhatsApp Message Delivery
AI reply is sent via the WhatsApp Cloud API using the tenant's Phone Number ID + access token. Message status updates (delivered, read) flow back via webhook and update the messages.status field.
📤
Send via WA Cloud APIPOST https://graph.facebook.com/v20.0/{phone_number_id}/messages. Type: text (or interactive for CTAs). Includes reply_to_message_id for threading.
✅
Delivery trackingWA Cloud API fires status webhooks: sent → delivered → read. Each updates messages.status in real-time. Failed deliveries trigger agent notification.
7
Meta Attribution + CRM Sync
Server-side events are fired to Meta Conversions API (CAPI) to feed the optimization algorithm and close the attribution loop. All data is simultaneously persisted to the CRM for the dashboard.
📊
Meta CAPI eventsLead event fired on first message. CompleteRegistration on qualification. Purchase on deal close. Each event carries event_id, phone hash, client_ip for dedup + matching.
🗄️
CRM updateLead stage, AI score, qualification_data, deal_value written to PostgreSQL. Activity log entry created for every state change. Analytics aggregation tables updated via DB triggers.
📡
Real-time dashboard pushWebSocket event emitted to connected dashboard sessions for this tenant. Agents see new leads, stage changes, and incoming messages in real-time without polling.
Human takeover protocol: Agents can take over any AI conversation from the dashboard inbox. When conversation.ai_enabled = false, the Conversation Worker skips AI processing and routes inbound messages directly to the agent's inbox. Agents can hand back to AI at any time via the toggle, which re-enables the worker and includes a context summary of the human conversation turn.
Claude system prompt structure: ROLE → PRODUCT_KNOWLEDGE (RAG) → QUALIFICATION_CRITERIA (BANT questions) → CONVERSATION_STAGE_INSTRUCTIONS → CLOSING_SCRIPT → OUTPUT_FORMAT_INSTRUCTIONS. Output format is always strict JSON. Temperature 0.3 for consistent, professional tone. Max tokens 512 per response to enforce concise WA-appropriate replies.
